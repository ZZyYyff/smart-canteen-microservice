# 需求规格说明书

## 1. 项目背景

随着高校和产业园区规模的扩大，传统食堂就餐模式面临诸多挑战：高峰期排队时间长、取餐秩序混乱、菜品供需信息不对称、商家备餐缺乏计划性。本项目旨在构建一套**智能食堂点餐与取餐微服务系统**，通过信息化手段提升食堂运营效率和用户就餐体验。

## 2. 系统目标

| 目标 | 描述 |
|------|------|
| 在线点餐 | 用户通过移动端浏览菜品、下单，减少现场排队 |
| 智能备餐 | 商家实时接收订单，按序制作，库存自动扣减 |
| 取餐排队 | 备餐完成后用户获取取餐号，大屏显示叫号信息 |
| 高效核销 | 用户凭取餐码扫码核销，防止错取冒领 |
| 统一网关 | 单一入口，JWT 认证，限流保护 |

## 3. 用户角色

| 角色 | 标识 | 描述 |
|------|------|------|
| 学生 | STUDENT | 浏览菜品、下单、查看订单、取餐核销 |
| 商家 | MERCHANT | 接单、制作、叫号、管理菜品和窗口 |
| 管理员 | ADMIN | 系统管理、用户管理、数据统计 |

## 4. 功能需求

### 4.1 用户服务

| 编号 | 功能 | 描述 |
|------|------|------|
| F-USER-01 | 注册 | 用户通过手机号、学工号、密码注册账号，手机号和学工号唯一 |
| F-USER-02 | 登录 | 支持手机号+密码登录，返回 JWT 双 Token（access + refresh） |
| F-USER-03 | Token 刷新 | 使用 refreshToken 换取新的 accessToken |
| F-USER-04 | 个人信息 | 查询和修改当前用户昵称、手机号 |
| F-USER-05 | 账号状态 | 支持 NORMAL / DISABLED 状态，禁用账号不可登录 |

### 4.2 菜品与菜单服务

| 编号 | 功能 | 描述 |
|------|------|------|
| F-MENU-01 | 菜品 CRUD | 商家新增、修改、删除菜品 |
| F-MENU-02 | 上下架 | 菜品 ON_SALE / OFF_SALE 状态切换 |
| F-MENU-03 | 库存扣减 | 原子扣减库存（UPDATE WHERE stock >= quantity），库存不足返回失败 |
| F-MENU-04 | 库存恢复 | 订单取消或下单失败时回滚库存 |
| F-MENU-05 | 低库存预警 | 库存低于预警阈值时标记 lowStock = true |
| F-MENU-06 | 每日菜单 | 创建日期+餐次（早/中/晚）+ 菜品关联的每日菜单 |
| F-MENU-07 | 今日菜单 | 按日期查询当前生效的每日菜单及菜品详情 |

### 4.3 订单服务

| 编号 | 功能 | 描述 |
|------|------|------|
| F-ORDER-01 | 创建订单 | 用户选择菜品和数量，系统逐项扣减库存后生成订单，失败则回滚已扣库存 |
| F-ORDER-02 | 订单状态流转 | CREATED → ACCEPTED → COOKING → WAIT_PICKUP → COMPLETED |
| F-ORDER-03 | 取消订单 | CREATED 或 ACCEPTED 状态可取消，取消后恢复库存 |
| F-ORDER-04 | 查询订单 | 按订单 ID 查询详情，按用户 ID 查询个人列表，按状态查询待处理 |
| F-ORDER-05 | 生成取餐信息 | 下单时生成随机 3 位取餐号（pickupNo）和 6 位取餐码（pickupCode） |

### 4.4 取餐与排队服务

| 编号 | 功能 | 描述 |
|------|------|------|
| F-PICKUP-01 | 窗口管理 | 新增窗口、查询列表、启用/停用窗口 |
| F-PICKUP-02 | 加入队列 | 订单进入 WAIT_PICKUP 后，order-service 调用本服务加入对应窗口 FIFO 队列 |
| F-PICKUP-03 | 叫号 | 商家点击叫下一号，选择 queue_time 最早的 WAITING 记录，状态改为 CALLED |
| F-PICKUP-04 | 大屏推送 | 叫号后通过 WebSocket 推送消息到食堂大屏（/ws/pickup/screen） |
| F-PICKUP-05 | 取餐核销 | 用户提供 pickupNo + pickupCode，校验成功后状态改为 FINISHED，并调用 order-service 完成订单 |

### 4.5 网关服务

| 编号 | 功能 | 描述 |
|------|------|------|
| F-GATEWAY-01 | 统一入口 | 所有 API 请求通过网关转发到对应微服务 |
| F-GATEWAY-02 | JWT 校验 | 除登录、注册、刷新 Token 外，所有接口校验 Bearer Token |
| F-GATEWAY-03 | 限流 | 基于 Redis 实现 IP 级和用户级限流，超出限制返回 429 |
| F-GATEWAY-04 | 用户信息传递 | Token 校验通过后将 userId 和 role 写入 X-User-Id、X-User-Role 请求头 |

## 5. 非功能需求

| 编号 | 类别 | 要求 |
|------|------|------|
| NF-01 | 可用性 | 微服务独立部署，单点故障不影响其他服务 |
| NF-02 | 可扩展性 | 通过 Nacos 服务发现支持动态扩缩容 |
| NF-03 | 可维护性 | 统一异常处理、统一返回格式、分层架构 |
| NF-04 | 数据一致性 | 库存扣减使用原子 SQL；下单失败自动回滚 |
| NF-05 | 响应时间 | 核心接口（下单、核销）响应时间 < 500ms |

## 6. 安全需求

| 编号 | 要求 | 实现方式 |
|------|------|---------|
| SEC-01 | 身份认证 | JWT（HMAC-SHA256），accessToken 24h 有效，refreshToken 7d 有效 |
| SEC-02 | 密码加密 | BCrypt 哈希存储，不可逆 |
| SEC-03 | 防止越权 | 网关校验 Token 后传递 userId，下游服务信任网关头 |
| SEC-04 | 防刷限流 | 网关层 Redis RateLimiter，默认 10 req/s，突发 20 |
| SEC-05 | 取餐码安全 | 6 位随机数字取餐码，核销时校验 pickupNo + pickupCode |

## 7. 性能需求

| 指标 | 目标值 |
|------|--------|
| 并发用户数 | 500 |
| API 响应时间 (P95) | < 1s |
| 下单成功率 | > 99.9% |
| 网关吞吐量 | > 1000 req/s |

## 8. 接口需求

### 8.1 外部接口

| 接口 | 协议 | 格式 | 说明 |
|------|------|------|------|
| REST API | HTTP/1.1 | JSON | 所有微服务对外接口 |
| WebSocket | WS | JSON | 大屏实时推送 |

### 8.2 内部接口

| 调用方 | 被调用方 | 接口 | 说明 |
|--------|----------|------|------|
| order-service | menu-service | PUT /api/menus/dishes/{id}/stock/deduct | 扣减库存 |
| order-service | menu-service | PUT /api/menus/dishes/{id}/stock/restore | 恢复库存 |
| order-service | pickup-service | POST /api/pickup/queue | 加入取餐队列 |
| pickup-service | order-service | PUT /api/orders/{id}/complete | 完成订单 |

### 8.3 统一返回格式

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

| 场景 | code | 说明 |
|------|------|------|
| 成功 | 200 | 操作成功 |
| 参数错误 | 400 | 请求参数不合法 |
| 未登录 | 401 | Token 缺失或无效 |
| 无权限 | 403 | 角色无权限 |
| 资源不存在 | 404 | 查询/操作的对象不存在 |
| 业务异常 | 1001 | 通用业务错误 |
| 订单状态异常 | 1002 | 状态流转不合法 |
| 取餐码错误 | 5001 | 核销码不匹配 |
| 已取餐 | 5002 | 重复核销 |
| 队列为空 | 5003 | 无排队记录 |
| 限流 | 429 | 超出请求频率限制 |
| 系统错误 | 500 | 服务器内部错误 |
